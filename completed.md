### Errors encountered + fixes

1. **Docker daemon not running.** `docker compose up -d` failed with "Cannot connect to the Docker daemon". Docker Desktop app was installed but not launched. **Fix:** `open -a Docker` + poll `docker info` until ready.

2. **`DATABASE_URL is not set` on `bun run db:smoke`.** The script called `loadEnvConfig(process.cwd())` at the top, but ESM hoists all `import` statements above top-level code — so `db/index.ts` read `process.env.DATABASE_URL` at import time, before `.env.local` was loaded. **Fix:** made `db/index.ts` lazy. Client + Drizzle instance are created on first property access via a `Proxy<Db>`, so env vars are read only when the first query runs. This is the correct pattern for any script that calls `loadEnvConfig` before importing `db`.

3. **Migration missing `CREATE EXTENSION` + HNSW indexes.** `drizzle-kit generate` doesn't know about pgvector extension setup or ANN indexes. **Fix:** hand-edit `db/migrations/0000_init.sql`. **Regenerating the migration will blow this away** — if you re-run `drizzle-kit generate` for schema changes, re-apply the extension line at top and the four HNSW `CREATE INDEX` lines at bottom.

4. **FashionCLIP Replicate version was fabricated.** First draft hardcoded a fake hash. **Fix:** made model slug + version env-driven (`REPLICATE_FASHIONCLIP_MODEL`, `REPLICATE_FASHIONCLIP_VERSION`); wrapper throws a clear error if the version env is empty. Phase 2 will need a real model picked from Replicate.

## Phase 2 — Vision Ingestion Pipeline (CODE COMPLETE, BLOCKED ON API KEYS 2026-04-18)

### What was built

- **Upload route** `app/api/ingest/upload/route.ts` — multipart POST accepting one or more files or a `.zip`. Extracts image entries via `adm-zip`, drops `__MACOSX/` + `._*` noise, writes each photo to `public/uploads/photos/<batchId>/<photoId>.<ext>`, inserts a `photos` row, creates a per-batch event bus, fires the worker (`void processBatch(...)`, not awaited), returns `{batch_id, accepted}`.
- **SSE stream route** `app/api/ingest/stream/route.ts` — GET `?batch_id=...`. Replays any backlog already emitted before the client connected, then streams live events, with 15s heartbeats and abort-signal cleanup. Closes when worker emits `batch_complete` / `batch_error`.
- **Manual merge route** `app/api/ingest/merge/route.ts` — POST `{garment_ids:[a,b]}`. Keeps the older row as canonical, reparents `garment_photos`, deletes the dropped row. Rejects cross-user merges.
- **Ingest pipeline modules** (all in `lib/ingest/`):
  - `events.ts` — `globalThis`-scoped `Map<batchId, {emitter, backlog, done}>`. Event shapes: `photo_accepted | photo_skipped | photo_processed | garment_created | garment_merged | batch_progress | batch_complete | batch_error`. Backlog lets late SSE subscribers catch up.
  - `storage.ts` — helpers for batch dirs (`ensureBatchDirs`), absolute paths, and `/uploads/...` public URLs for both photos and crops.
  - `filter.ts` — `containsClothing(bytes, mime)` via Gemini 2.5 Flash (temperature 0, maxOutputTokens 4, yes/no). On Gemini failure defaults to **true** (prefer false inclusions over dropping real photos).
  - `extract.ts` — batched Gemini 2.5 Flash garment extraction with `responseSchema` JSON mode. 3-shot examples in the system prompt. Zod-validated, `confidence >= 0.4` filter, `photo_index` range check. Single retry on parse failure.
  - `embed.ts` — `cropGarment` with `sharp` (honors EXIF rotation, flattens alpha, sRGB, JPEG q88), then `embedCrop` (Replicate FashionCLIP, 512-dim check) and `embedDescription` (Gemini text-embedding-004, 768-dim check).
  - `dedup.ts` — kNN cosine query over `garments` where `user_id = ? AND category = ?`, threshold `DEDUP_DISTANCE_MAX = 0.12` (= 0.88 similarity; tunable constant). Returns top match or null. `attachPhotoToGarment` writes a `garment_photos` row and optionally promotes the new crop to hero.
  - `worker.ts` — orchestration: filter (concurrency 4) → batched extract (5 photos/call) → per-garment crop + FashionCLIP + dedup → either merge or insert (+ text embedding). Emits events at every phase. Wraps fatal errors into `batch_error`.
- **UI page** `app/ingest/page.tsx` + `app/ingest/ingest-client.tsx` — click-or-drag file drop, `EventSource` subscription to the SSE endpoint, live-appending garment grid (hero crop + category + brand guess), progress line `Processed N/M photos, K unique garments`.

### Errors encountered + fixes (Phase 2)

1. **`SchemaType.STRING` + `enum` required `format: "enum"`** in `@google/generative-ai`'s `ResponseSchema` type. Added `format: "enum"` to the `category` property in `extract.ts`. Without it, TS rejects the schema as not matching `EnumStringSchema`.

2. **`unzipper` pulls in `@aws-sdk/client-s3` as an optional peer**, and Turbopack's module resolver fails on the unconditional `require('@aws-sdk/client-s3')` inside `unzipper/lib/Open/index.js` even when you never call the S3 source. Surface symptom was `/ingest`, `/api/ingest/upload`, and `/api/ingest/stream` all returning 500 with "Module not found: Can't resolve '@aws-sdk/client-s3'". **Fix:** swapped to `adm-zip` (pure JS, no peers). If we ever need streaming zip support, consider `yauzl` instead.

### Context future sessions need

- **Worker lives in-process** behind `globalThis._caasIngestBuses`. In dev, Turbopack HMR can kill the worker mid-batch — this is acceptable for the hackathon. Don't rely on batches surviving a reload.
- **SSE replay** is backlog-based: the event bus keeps every emitted event for the batch's lifetime so a client that connects after the upload-POST returns still catches up. Memory will grow if batches get huge — fine for demo, not production.
- **Dedup threshold** `DEDUP_DISTANCE_MAX = 0.12` (cosine distance, inverse of the TODO's `0.88` similarity). Tune in `lib/ingest/dedup.ts` during rehearsal — it's a single constant per the TODO instructions.
- **Hero crop replacement** is currently NEVER done on merge (we pass `newCropBeatsHero: false`). The TODO says "keep higher-confidence crop as hero" — we'd need to store `extraction_confidence` alongside the hero to compare. Deferred; revisit in Phase 3 or during rehearsal if hero crops look bad.
- **HEIC handling** is untested. `sharp` supports HEIC only with libvips built with libheif; the Replicate call won't care if the public URL is `.heic`. If Safari users upload HEIC, this may break — consider converting to JPEG at upload time if it becomes an issue.
- **Public uploads dir** — crops and photos both live under `public/uploads/`. Add `.gitkeep` + ignore rule if we start committing. Currently un-ignored; will bloat the repo if committed.
- **FashionCLIP output dim** is asserted to be 512. If a non-FashionCLIP model is swapped in with different dims, the `garments.image_embedding` column type needs to match — and a schema migration is required. Don't change the underlying model without checking dims.
- **FashionCLIP service contract:** local FastAPI service at `FASHIONCLIP_URL` (default `http://localhost:8001`) exposing `POST /embed/image` and `POST /embed/text`, both accepting JSON `{image: base64}` / `{text}` and returning `{embedding: number[]}`. Our TS client (`lib/clients/fashionclip.ts`) and `embedCrop` (`lib/ingest/embed.ts`) assume this shape — changing the Python side's request/response shape means updating both here.
- **FashionCLIP install layout:** everything lives under `fashionclip/`. Committed to git: `service.py`, `requirements.txt`, `start.sh`, `.gitignore`. Gitignored: `venv/`, `hf-cache/`, `__pycache__/`. `start.sh` sets `HF_HOME="$PWD/hf-cache"` so model weights (~600MB) stay inside the project dir rather than polluting `~/.cache/huggingface`.
  - **First-time setup:** `cd fashionclip && python3 -m venv venv && source venv/bin/activate && pip install -r requirements.txt`, then `./start.sh` (auto-downloads the model on first run).
  - **Run each session:** `./fashionclip/start.sh` from the repo root.
  - **Nuke everything reclaimable (~2.5GB):** `rm -rf fashionclip/venv fashionclip/hf-cache`. The committed code stays. Nuke the whole dir if you also want the code gone.
- **CLIP text embeddings are available but unused by default.** `embedTextClip` is exported for completeness but the garment schema uses Gemini `text-embedding-004` (768-dim) for descriptions, not CLIP text (512-dim). Don't mix — they live in different embedding spaces and kNN won't be meaningful across them.
